# Transmission Control Protocol (TCP)
- **Reliable** transport between sending and receiving process
- **Flow control**: Sender won't overwhelm receiver
- **Congestion control**: Throttle sender when network overloaded
- Does not provide
	- Timing
	- Minimum throughput guarantee
	- Security
- Connection-Oriented: Setup required between client and server processes

## Connection-Oriented Protocol
- Why do we need connection setup?
	- To establish **state** on both hosts
	- Most important state: sequence numbers
		- Count the number of bytes that have been sent
		- Initial value chosen at random. Why?
			- Security reasons. This practice helps mitigate the risk of certain types of attacks such as sequence number prediction attacks and connection hijacking

# Important TCP Flags
- TCP (Transmission Control Protocol) uses a set of flags within its header to control and manage various aspects of the communication  between two devices
- There are 6 control flags
- Important Flags
	- **SYN**: synchronization, used for connection setup
	- **ACK**: acknowledge received data
	- **FIN**: finish, used to tear down connection

# Analyzing TCP Connections with Wireshark

# Sequence Number Space

# Retransmission
- Packets could be lost or damaged
	- This leads to a lack of acknowledgment
	- The sender will resend the packets
- When to resend?
	- Use a timer when sending a packet
	- Lack of ACK will trigger the timer to timeout
		- Retransmit the packet when the time expires
## Retransmission Scenario 1
- When the data packet is lost or erroneous

## How Long Should the Sender Wait?
- Scenarios 2 and 3 could be avoided to improve efficiency
	- Set a proper timer. How?
- **Too Short**: Premature timeout, unnecessary retransmissions
- **Too long**: Slow reaction to segment loss: Excessive delay when the packet is lost
- TCP sets the timeout as a function of the RTT (round trip time)
	- **RTT**: The time required for the packet to travel from the source to the destination and then come back again
	- RTT varies depending on the connection
### How To Estimate RTT?
- `SampleRTT`: Measured time from segment transmission until ACK receipt
- `SampleRTT` will vary, want estimated RTT "smoother"
	- Average several recent measurements, not just current `SampleRTT`
- Problem: Timeout period can be relatively long or too short
	- When a segment is lost, this long timeout period forces the sender to delay resending the lost packet
## Problem with RTT-Based Strategy
![[Pasted image 20260302151416.png|350]]
- There is an ambiguity of not knowing whether the acknowledgment is for the original transmission or for the retransmission
- Whenever the retransmission occurs, it starts sending from the lost packet onward
	- This is inefficient

# Fast Retransmission
- TCP fast retransmit
	- If sender receives 3 additional ACKs for same data ("triple duplicate ACKs"), resend unacknowledged segment with smallest seq #
		- Likely that unacknowledged segment lost, so don't wait for timeout
![[Pasted image 20260302152449.png|300]]
- Receipt of three duplicate ACKs indicates 3 segments received after a missing segment - lost segment is likely, so retransmit
## What Will Happen After the Retransmission?
- Many TC implementations will buffer **correctly received but out-of-order** segments
- A proposed modification to TCP allows a TCP receiver to acknowledge out-of-order segments selectively rather than just cumulatively acknowledge the last correctly received, in order segment
## What Should the Receiver ACK?
- **Cumulative ACK**
	- Where an ACK for sequence *n* implies ACKS for all *k < n*
	- Possible problem
		- Receiver will hold **correctly received but out-of-order** in the buffer and sender may even retransmit them
- **Selective ACKs (SACKSs)**
	- Indicate those that did arrive, even if not in order
	- An actual TCP extension
	- Require both sender and receiver support the SACK feature
## Selective ACK (SACK)
- SACK does not replace the original ACKs but adds another field in the TCP header's options field for SACK information
	- A maximum of 32 bytes (out of 40 bytes) can be used for SACK in the 'options' field. 8 bytes in the 'Options' field are already reserved for 'timestamps'
![[Pasted image 20260304140327.png|350]]
### Retransmission with SACK
- SAC feature helps the sender to identify 'gaps' in the receiver buffer
- SACK helps the sender to know about these holes in the receiver buffer at an early time
	- When the sender knows about the holes or lost packets, it retransmits them at the same time
- Note the ACK for the last packet
	- The ACK = 7, not 4
	- It 'fills' the gap
![[Pasted image 20260304140654.png|450]]

# Flow Control
- To improve the efficiency, the packets should not be sent one by one (wait for the acknowledge before sending the next one)
- Problem: how many packets should a sender transmit?
	- **Pipelining**: Multiple packets can be sent and acknowledged at the same time. Allow more efficient use of the network connection and faster transfer
	- Too many packets may overwhelm the receiver
	- Size of the receivers buffers may change over time
## Window Size
- Solution: **Sliding Window**
	- Receiver tells the sender how big their current buffer is
	- Called the *advertised window/receive window*
	- For window size *n*, sender may transmit *n* bytes **without receiving an ACK**
	- After each ACK, the window slides forward
- The window is not a static size
	- Window may go to zero
		- When the buffer is full
![[Pasted image 20260304143108.png|350]]
- What is the range of the receive window?
	- 16-bits integer: 0 to 65535
- The range is relatively small. How to specify a large receive window size?
	- Using "options" field to specify the scaling factor
	- When? In the handshake process
- In Wireshark, the true size is specified in the "Calculated window size" value
![[Pasted image 20260304143310.png|350]]
## What to Do When The Buffer Is Full?
- This window size is not static
	- Can go to zero when buffer is full
- Sender will keep sending 1 -byte segments asking if the window size comes back > 0 until it does

# Observations
- Throughput is ~ w/RTT (approximately proportional to the reciprocal of the round-trip time)
	- As the RTT increases, the throughput decreases, and vice versa
- Sender has to buffer all *unacknowledged packets*, because they may require retransmission
- Receiver may be able to accept out-of-order packets, but only up to buffer limits
## Congestion Control
- What is congestion?
	- Informally: too many sources sending too much data too fast for network to handle
- Different from flow control
![[Pasted image 20260304143704.png|250]]
![[Pasted image 20260304143720.png|250]]
### What Is Congestion?
- Load on the network is higher than capacity
	- Capacity is not uniform across networks
		- Modem vs Cellular vs Cable vs Fiber Optics
	- There are multiple flows competing for bandwidth
		- Residential cable modem vs corporate datacenter
	- Load is not uniform over time
		- 10pm, Sunday night = BitTorrent Game of Thrones
![[Pasted image 20260304143949.png|400]]
## Advertised Window, Revisited
- Does TCP's advertised window solve congestion?
	- ***NO***
- The advertised window only protects the receiver
- A sufficiently fast receiver can max the window
	- What if the network is slower than the receiver?
	- What if there are other concurrent flows?
- Key points
	- Need a window size determines send rate
## Problems When Congestion Occurs
- Results in **packet loss**
	- Routers have finite buffers
	- Internet traffic is self similar, no buffer can prevent all drops
	- When routers get overloaded, packets will be dropped
- **Increases Delay**
	- Router queues build up
	- Wasted bandwidth from **retransmissions**
## Congestion Control Approaches
- **End-End congestion control**
	- No explicit feedback from network
	- Congestion *inferred* from observed loss, delay
	- Approach taken by TCP
![[Pasted image 20260304144508.png|400]]
- **Network Assisted congestion control**
	- Routers provide direct feedback to sending/ receiving hosts with flows passing through congested router
	- May indicate congestion level or explicitly set sending rate
	- Approach taken by TCP Explicit Congestion Notification (ECN)
![[Pasted image 20260304144457.png|400]]

# TCP Congestion Control
- Each TCP connection has a window
	- Controls the number of unACKed packets
- Idea: vary the window size to control the send rate
- Introduce a *congestion window (cwnd)* at the sender
	- Congestion control is sender-side problem
## Two Basic Components
1. Detect Congestion
	- Packet dropping is most reliable signal
	- How do you detect packet drops?
		- Timeout after not receiving an ACK
		- Several duplicate ACKs in a row (fast retransmission)
2. Rate Adjustment Algorithm
	- Modify *cwnd*
	- Probe for bandwidth
	- Responding to congestion
## How to Adjust Rate?
- Basic Algorithm
	- Upon receipt of ACK: increase cwnd
		- Data was delivered, perhaps we can send faster
		- cwnd growth is proportional to RTT
	- On loss: decrease cwnd
		- Data is being lost, sign of congestion
- Question: increase/ decrease functions to use?
## Implementing Congestion Control
- Maintains three variables
	- ***cwnd***: Congestion Window
	- ***adv_wnd***: Receiver Advertised Window (discussed in flow control)
	- ***ssthresh***: Slow Start Threshold Size (used to update cwnd)
- For sending, use `wnd = min(cwnd, adv_wnd)`
- Two phases of congestion control
	1. Slow Start (cwnd < ssthresh)
		- Probe for bottleneck bandwidth
	2. Congestion avoidance (cwnd >= ssthresh)
		- AIMD (Additive Increase, Multiplicative Decrease)
