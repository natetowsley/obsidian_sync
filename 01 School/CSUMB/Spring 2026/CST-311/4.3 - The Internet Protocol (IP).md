# Time-to-Live (TTL)
- TTL is deployed as a **counter** embedded in each packet. When the predefined timespan or event count expires, the packet is either discarded or revalidated
	- Prevent data packets from moving across the network indefinitely
- Administrators can set TTL to any value between 1 and 255
- When a packet is processed by a router, the router subtracts 1 from the TTL count. Then, the packet travels to the next destination on the network
- When the TTL count reaches 0, after the final subtraction, the packet is discarded by the router

# Fragmentation
- **Problem**: Each network has its own maximum transmission unit (MTU)
	- Networks allowed to be heterogeneous
	- Minimum MTU may not be known for a given path
- **IP Solution: Fragmentation**
	- Split datagrams into pieces when MTU is reduced
	- Reassemble original datagram at the receiver
![[Pasted image 20260406144031.png|450]]
## IP Header
- **Identifier**: a unique number for the original datagram
- **Flags**: Consists of a 3-bit field of which the two low-order (least-significant) bits control fragmentation
	- The left-most bit is reserved (and must be 0, i.e. Must Be Zero (MBZ))
	- The middle bit specifies whether the packet can be fragmented (0 = may fragment, 1 = do not fragment).
		- If a datagram cannot be fragmented but exceeds the MTU for the local network then the router will drop the packet upon arrival
	- The right-most bit (M) specifies whether the packet is the last fragment in a series of fragmented packets (M = 0: last fragment, M = 1: more fragments coming)
- **Offset**: Byte position of the first byte in the fragment
	- Indicates the position of the fragment's data relative to the beginning of the data in the original datagram, which allows the destination IP process to properly reconstruct the original datagram
	- Initial offset = 0, next offset = current offset + size of current packet

## IP Fragment Reassembly
- Performed at destination
- M = 0 fragment gives us total data size
	- 360 - 20 + 3460 = 3800
- Challenges:
	- Out-of-order fragments
	- Duplicate fragments
	- Missing fragments
- Basically, memory management nightmare
![[Pasted image 20260406144756.png|300]]
## Fragmentation Concepts
- Highlights many key Internet characteristics:
	- **Decentralized and heterogeneous**
		- Each network may choose its own MTU
	- **Connectionless datagram protocol**
		- Each fragment contains full routing information
		- Fragments can travel independently, on different paths
	- **Best effort network**
		- Routers/receiver may silently drop fragments
		- No requirement to alert the sender
	- **Most work is done at the endpoints**
		- I.E. Reassembly
## Fragmentation in Reality
- Fragmentation is expensive
	- Memory and CPU overhead for datagram reconstruction
	- Want to avoid fragmentation if possible
- MTU discovery protocol
	- Send a packet with "don't fragment" bit set
	- Keep decreasing message length until one arrives
	- May get "can't fragment" error from a router, which will explicitly state the supported MTU
- Router handling of fragments
	- Fast, specialized hardware handles the common case
	- Dedicated, general purpose CPU just for handling fragments

# IP Addressing
- **IP address**: 32-bit identifier associated with each host or router **interface**
- **Interface**: Connection between host/router and physical link
- Router's typically have multiple interfaces
- Host typically has one or two interfaces (e.g., wired Ethernet, wireless 802.11)
![[Pasted image 20260406145854.png|350]]
## IP Address Structure
![[Pasted image 20260408140026.png|305]]
- Two (2) main parts:
	- **Subnet part**: Devices in same subnet have common high order bits
	- **Host part**: Remaining low order bits

# Subnets
- What's a subnet?
	- Device interfaces that can physically reach each other without passing through an intervening router
	- There are 3 subnets in the network above
### Subnets: Example
- Consider one of the subnets:
	- IP 1: 223.1.3.141
		- `11011111 00000001 00000011 1001101`
	- IP 2: 223.1.3.1
		- `11011111 00000001 00000011 00000001`
	- IP 3: 223.1.3.2
		- `11011111 00000001 00000011 00000010`
- What's in common?
	- First 24 bits - subnet part
- Last 8 bits - host part
![[Pasted image 20260408140405.png|300]]
## Defining Subnets
- Recipe for defining subnets:
	- **Detach** each interface from its host or router, creating "islands" of isolated networks
	- Each isolated network is called a subnet
![[Pasted image 20260408140457.png|300]]
## Subnet Mask
- Subnet mask: /24?
	- IP addresses in this same subnet will match the same first 24 bits
	- Does this sound familiar?
		- Longest prefix matching in the last section
### Subnet Mask Representation
- Using asterisk (`*`) notation is not a good idea
- Better way:
	- Use a standard IPv4 address (e.g., 223.1.3.141)
	- And a 32-bit of `0` and `1` subnet mask (e.g., 255.255.255.0)
	- The `1` bits in the subnet mask identify the network (subnet) portion of the IP address, while the `0` bits represent the host portion
- Example
	- IP address: 223.1.3.141
	- Subnet mask: 255.255.255.0
	- This combination specifies that the subnet includes all IP addresses that share the same network portion (223.1.3.x) and have varying host addresses (from 0 to 255)
## CIDR
- **CIDR**: Classless Inter Domain Routing (pronounced "cider")
	- Subnet portions of address of arbitrary length
	- Address format:
		- `a.b.c.d/x`
		- where `x` is the number of bits in t he subnet portion of the address
![[Pasted image 20260408141102.png|350]]
- EX:
	- IP address: 223.1.3.141
	- Subnet mask: 255.255.255.0
	- CIDR:
		- 223.1.3.141/24
![[Pasted image 20260408141357.png|300]]
- ***CIDR BLOCK SIZES MUST BE A POWER OF 2***
# How to Get IPs?
- There are two questions:
	- Q: How does a network get IP address for itself (network part of address)?
	- Q: How does a host get IP address within its network (host part of address)
## How Does a Network Get IP Address for Itself?
- IP address ranges controlled by IANA, which is now part of ICANN (Internet Corporation for Assigned Names and Numbers)
- IANA grants IPs to regional authorities
- ISPs obtains an address block, and subdivide to clients:
- EX:
	- ISP: 128.35.16/20         ***10000000 00100011 0001***0000 00000000
	- Client 1: 128.35.16/22 ***10000000 00100011 000100***00 00000000
	- Client 2: 128.35.20/22 ***10000000 00100011 000101***00 00000000
	- Client 3: 128.35.24/21 ***10000000 00100011 00011***000 00000000

# How Does a Host Get IP Address?
- First Approach: manually
	- Hard-coded by sysadmin in config file (e.g., /etc/rc.config in UNIX)
	- Tedious and error-prone: e.g, laptop going from BIT to library to OSU
- Solution: Dynamic Host Configuration Protocol (DHCP)
	- **Dynamically** get address from server
	- "plug-and-play" approach

# DHCP: Dynamic Host Configuration Protocol
- Client-Server protocol
	- Typically, DHCP **server** will be co-located in router, serving all subnets to which router is attached
	- When joining a network, a **host** (client) will request an address from the server
	- The protocol allows reuse of addresses and support for mobile users who join or leave the network
- An **application-layer** protocol
	- Uses **UDP** as its transport protocol
- Overview:
	- Host broadcasts **DHCP discover** message (optional)
	- DCHP server response with **DHCP offer** message (optional)
	- host requests IP address: **DHCP request** message
	- DHCP server sends address: **DHCP ack** message
## DHCP Client-Server Scenario
![[Pasted image 20260408142707.png|400]]
### DHCP Discover Message
- Is there a DHCP server out there?
- This message is broadcasted to every host and routers in the subnet
![[Pasted image 20260413141323.png|300]]
### DHCP Offer Message
- **Broadcast**: I'm a DHCP server! Here's an IP address you can use
- Offer an IP address to the client
![[Pasted image 20260413141515.png|300]]
### DHCP Request Message
- **Broadcast**: Okay! I would like to use this IP address
- Request to use the offered IP address
- Note: *Previous steps are optional. The client could request to use the current assigned IP address, or an IP address from before*
![[Pasted image 20260413141642.png|300]]
### DHCP ACK Message
- **Broadcast**: Okay great! You've got the IP address
- Acknowledgement of the new IP address for the arriving client
![[Pasted image 20260413141755.png|300]]
## Why All Messages Are Broadcast in DCHP?
- There could be more than one DHCP server in the subnet
- The broadcast ensures that all the responding DHCP servers know that the client has chosen a server
- The servers that are not chosen can cancel the reservations for the IP addresses that they had offered
	- The selected server allocates the IP address for the client and stores the information in the DHCP data store
## DHCP: More Than IP Addresses
- DHCP can return more than just allocated IP address on subnet:
	- Address of first-hop router for client
	- Name and IP address of DNS server
	- Network mask (indicating network versus host portion of address)

# Network Address Translation (NAT)
- Despite CIDR, it is still difficult to allocate addresses
	- $2^{32}$ is only 4.3 billion
- We will talk about IPv6 later
- NAT "hides" entire network behind one address
	- Hosts are given **private addresses**
	- Routers map outgoing packets to a free address/port
	- Router revers maps incoming packets
- All devices in local network share just one IPv4 address as far as the outside world is concerned
![[Pasted image 20260413142250.png|550]]
- All devices in local network have 32-bit addresses in a "private" IP address space with one of the following prefixes:
	- `10/8 (10.0.0.1, 10.0.0.2, ...)`
	- `192.168/16 (192.168.86.68, 192.168.1.1, ...)`
	- `172.16/12 (172.16.1.2, ...)`
	- Only be used in local network
### NAT: Private IP Address
- How about **`127.0.0.1`**? Is it a private address?
- A special-purpose IPv4 address and is called the **localhost** or **loopback** address
	- All computers use this address as their own, but NOT for communication with other devices
	- Application software typically uses this loopback for local testing purposes
### NAT: Advantages
- Just *one* IP address needed from provider ISP for all devices
- Can change addresses of host in local network without notifying outside world
- Can change ISP without changing addresses of devices in local network
- Security: devices inside local net not directly addressable, visible by outside world
### NAT: Implementation
- NAT router must (transparently),
	- for outgoing diagrams, replace (source IP address, port #) of every outgoing datagram to (NAT IP address, new port #)
	- remote clients/servers will respond using (NAT IP address, new port #) as destination address
	- remember (in NAT translation table) every (source IP address, port #) to (NAT IP address, new port #) translation pair
	- for incoming datagrams: replace (NAT IP address, new port #) in destination fields of every incoming datagram with corresponding (source IP address, port #) stored in NAT table

### NAT: Multiple Web Servers Behind a NAT
- To host a web server or any service that will be accessed from the public Internet behind a NAT/router, you will need to have a **port-forwarding table** set up to handle incoming packets for different services
	- If your NAT doesn't have port-forwarding table set up, no services run behind the NAT can be accessed from outside

### NAT: Some Problems
- NAT has been controversial:
	- Routers "should" only process up to layer 3
	- Address "shortage" should be solved by IPv6
	- NAT traversal: what if client wants to connect to server behind NAT?
- but NAT is here to stay:
	- Extensively used in home and institutional nets, 4G/5G cellular nets

# IPv6
- 128-bit addresses
- Address format
	- 8 groups of 16-bit values, separated by `:`
	- Leading zeroes in each group may be omitted
	- **Consecutive** groups of zeroes can be omitted using `::`
		- Should only be used once in an IPv6 address
![[Pasted image 20260413143746.png|350]]
- What is the localhost (127.0.0.1) in IPv6?
	- `::1`
## IPv6: Datagram Format

![[Pasted image 20260413143858.png|550]]
### Flow Label
- Used by a source to label the packets belonging to the same flow in order to request special handling by intermediate IPv6 routers, such as non-default quality of service or real-time service
- The concept of "flow" is not well defined
	- Policy up to the ISPs
	- IPv6 only provides the mechanism, not the policy
### IPv6: Differences from IPv4 Header
- Missing header length - rolled into Next Header field
- No checksum
- Do not support fragmentation/reassembly
	- Hosts are expected to use path MTU discovery
- No options
### IPv6: Improvements
- Expanded address space
- Improve addressing and address autoconfiguration
	- No more manual configuration or the use of NAT
- Speed:
	- No checksum to verify
	- No need for routers to handle fragmentation
- Better security and quality of service features
## Deployment Challenges
![[Pasted image 20260413144243.png|450]]
- Switching to IPv6 is a whole-Internet upgrade
	- All routers, all hosts
	- ICMPv6, DHCPv6, DNSv6
- Not all routers can be upgraded simultaneously
	- How will network operate with mixed IPv4 and IPv6 routers
## Transitioning to IPv6
- Today, most network edges are IPv6 ready
	- Windows/OSX/iOS/Android all support IPv6
	- Your wireless access point probably supports IPv6
- The internet core is hard to upgrade
	- but an IPv4 core cannot route IPv4 traffic
![[Pasted image 20260413144610.png|400]]
### Transitioning Technologies
- How do you route IPv6 packets over and IPv4 Internet?
- Transition Technologies:
	- Use tunnels to encapsulate and route IPv6 packets over the IPv4 Internet
	- Several different implementations:
		- 6to4
		- IPv6 Rapid Deployment (6rd)
		- Teredo
		- ... etc.
#### Tunneling and Encapsulation
- **Tunneling:** IPv6 datagram carried as payload in IPv4 datagram among IPv4 routers ("packet within a packet")
- Tunneling used extensively in other contexts (4G/5G)
